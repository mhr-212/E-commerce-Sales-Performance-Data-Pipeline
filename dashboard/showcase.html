Code

Key Patterns

Two production patterns that separate this pipeline from typical tutorials.

load.py — Checksum-Gated UPSERT
# Zero I/O on re-runs — only writes changed rows INSERT INTO fact_sales ( sale_id, sale_date, product_sk, net_amount, row_checksum, ... ) VALUES (:sale_id, :sale_date, ...) ON CONFLICT (sale_id, sale_date) DO UPDATE SET net_amount = EXCLUDED.net_amount, row_checksum = EXCLUDED.row_checksum WHERE fact_sales.row_checksum != EXCLUDED.row_checksum; # ↑ Fully idempotent — safe to re-trigger # any past date with zero DB side effects
extract.py — Schema Contract
# Pandera catches bad rows at the boundary SALES_SCHEMA = DataFrameSchema({ "sale_id": Column(str, unique=True), "quantity": Column(int, Check.gt(0)), "discount_pct": Column( float, Check.in_range(0.0, 1.0) ), "channel": Column(str, Check.isin(["online", "mobile_app", "marketplace", "in_store"]) ), }, coerce=True) # Bad rows → dead_letter/, not dropped silently # ↑ Catches upstream issues at source, not # silently 3 steps later in the warehouse

Innovation

What Makes This Top-1%

These patterns separate production data engineering from tutorials.

🔒
Checksum-Gated UPSERT
Row-level SHA-256 checksum stored in the warehouse. WHERE row_checksum != EXCLUDED.row_checksum means zero disk I/O on re-runs. Fully idempotent.
PostgreSQL
🚦
Pandera Schema Contracts
DataFrameSchema validates column types, ranges, and allowed values at the extract boundary. Bad rows routed to dead-letter with error annotation — never silently dropped.
Data Quality
🧩
SCD Type-2 Products
valid_from / valid_to on dim_product tracks price changes over 5 years without corrupting historical revenue accuracy.
Dimensional Modeling
📊
Concurrent Mart Refresh
REFRESH MATERIALIZED VIEW CONCURRENTLY — BI dashboards never experience downtime or stale reads during refresh cycles.
Zero Downtime
🗂️
Monthly Range Partitioning
fact_sales partitioned by month. PostgreSQL partition pruning skips 90%+ of data on typical date-range analytics queries — stays fast at 50M+ rows.
Scalability
🔍
ContextVar Trace Logging
Python contextvars injects run_id + task_id + dag_id into every log line — full trace correlation across distributed tasks without thread-safety issues.
Observability
✔️
validate_load Gate
SUM(net_amount) parity check between source CSV and database after every load. Raises AssertionError if variance > 0.1% — hard stop on data loss.
Data Integrity
🔄
all_done Cleanup Task
Airflow trigger_rule="all_done" ensures temp Parquet staging files are removed even when upstream tasks fail — no orphaned files accumulating in production.
